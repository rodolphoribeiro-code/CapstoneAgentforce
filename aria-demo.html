Authentication design decision

✅ New reservation

No authentication required. The customer provides their name and email at booking time. Goes directly to create_reservation.

Rationale: a new customer has no existing account to verify against. Requiring auth for a first booking creates unnecessary friction.

🔐 Modify / Cancel

Requires OTP verification. The customer provides their email, receives a 6-digit code, and enters it before accessing reservation management.

Rationale: modifying or cancelling a reservation affects existing data tied to a specific contact — identity must be confirmed.

Agent Script — Annotated Source

Agent Script is Salesforce's DSL for the Atlas Reasoning Engine. It is NOT JavaScript or Python. Each block has a specific role in the two-phase execution model.








  
Session Variables

  
Variables persist across the entire conversation session. They are the agent's memory — subagents read and write them to share state.


  
💡 Mutable variables have a default value and can be read and written by the agent. Linked variables are read-only — they are populated automatically from the messaging session context (e.g., the user's ContactId).

Apex Actions

  
Each action is an @InvocableMethod Apex class. The agent calls them via target: "apex://ClassName". All use Database.queryWithBinds with SYSTEM_MODE.


  
⚠️ For the demo, the OTP code is fixed at 123456 in VerifyOtpCode.cls. Replace with a real email delivery mechanism for production.


  







  
Dining Policies

  
All policies are embedded directly in the faq subagent's instructions: -> block. The LLM reads them from the resolved prompt in Phase 1 — no Apex action or database lookup required.


  
💡 Why static instructions? For a demo, hardcoding policies in the agent is the fastest path. For production, a GetDiningPolicy Apex action backed by a custom object would allow policies to be updated in Salesforce without redeploying the agent.

How it works in Agent Script

  
subagent faq:
    description: "Answers restaurant info AND policy questions..."
    reasoning:
        instructions: ->
            | Help the customer with restaurant information or policy questions.
            | Use {!@actions.get_restaurant_details} for specific restaurant info.
            // ...restaurant action instructions...

            --- THE DINER POLICIES ---

            CANCELLATION POLICY:
            | Reservations may be cancelled free of charge up to 24 hours before booking.
            | Cancellations under 24h may incur a $15 per guest no-show fee.
            // ... more policy text ...

            GROUP BOOKING POLICY:
            | Groups of 8+ require a reservation at least 48 hours in advance.
            // ...


  
⚠️ The LLM answers policy questions purely from this prompt text — it does NOT call any Apex action. This means policies cannot be updated without redeploying the agent. Consider a GetDiningPolicy Apex action backed by a DiningPolicy__c custom object for production.

  • Captures outputs with set @variables.x = @outputs.y

  • Applies before_reasoning and after_reasoning hooks

  • Transitions deterministically if conditions are met

  • Receives the fully resolved prompt from Phase 1
