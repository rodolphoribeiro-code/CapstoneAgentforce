Overview

This agent was built as the capstone for the FDE "Ready in Six" (R6) Onboarding Program. The use case is an Online Hospitality Platform where guests interact with an AI-powered support agent to manage their restaurant reservations.


The agent runs on the Atlas Reasoning Engine (Salesforce's native LLM orchestration layer) and is defined entirely in Agent Script — a declarative scripting language for authoring Agentforce agents. All business logic is implemented in Invocable Apex classes that run in SYSTEM_MODE to bypass field-level security restrictions for the Einstein Agent User.

Subagent Map

The agent uses a Hub-and-Spoke architecture. Every conversation starts at agent_router, which routes to a specialized subagent. Each subagent can transition back to the hub or escalate to a human.

agent_router (start)
Entry point. Classifies intent and routes to the correct subagent. Never executes actions itself.
create_booking
Collects restaurant name, date/time, party size, and customer name, then calls CreateBooking Apex. Uses require_user_confirmation for platform-native confirmation.
cancel_booking
Collects booking number, then calls ModifyOrCancelBooking with action=CANCEL. Platform confirmation via require_user_confirmation: True.
modify_booking
Collects booking number + new date/time and/or party size, confirms, then calls ModifyOrCancelBooking with action=MODIFY.
restaurant_info
Collects restaurant name, calls GetRestaurantDetails, returns full details from the database.
escalation
Transfers to a human agent via @utils.escalate when the user requests one or the issue is complex.
off_topic
Politely declines requests outside dining/reservations. Never answers general knowledge questions.
ambiguous_question
Asks for clarification when the user's intent is unclear or doesn't match any topic.

Key Design Decisions

Why SYSTEM_MODE? The Einstein Agent User is a special Salesforce user that runs agent actions. By default, Apex runs in user mode, which applies Field-Level Security (FLS). The Einstein Agent User's FLS profile does not grant access to custom text fields like Location__c, Phone__c, and Opening_Hours__c — causing those fields to return null even though the data exists. Switching to Database.query(..., AccessLevel.SYSTEM_MODE) bypasses FLS and ensures all fields are readable. Object-level access is still controlled by the Permission Set.
Why require_user_confirmation: True? For destructive or irreversible actions (cancel, create), the action definition includes require_user_confirmation: True. This delegates the confirmation prompt to the platform runtime — the LLM does not need to ask "are you sure?" in text, which was causing a pattern where the LLM confirmed verbally but then never called the action. Platform-native confirmation is more reliable and produces a consistent UI.

How to Test

Prerequisites: Salesforce CLI (sf) installed, authenticated to the DinerAgent org, and working directory set to ~/Projects/CapstoneAgentforce.

Option A — CLI Preview (Programmatic)

Best for testing specific utterances and verifying action invocation. Two commands: start creates a session, send sends messages.

Navigate to the project

cd ~/Projects/CapstoneAgentforce

Start a preview session with live actions

sf agent preview start \
  --use-live-actions \
  --authoring-bundle Intelligent_Diner_Support_Agent \
  --target-org DinerAgent \
  --json

Copy the sessionId from the JSON output — you'll use it in every send command.

Send a message (flag is -u, not --utterance)

sf agent preview send \
  --authoring-bundle Intelligent_Diner_Support_Agent \
  --session-id <YOUR_SESSION_ID> \
  -u "Tell me about La Bella Italia" \
  --target-org DinerAgent \
  --json

The agent's response appears under result.messages[0].message in the JSON output.

Verify database changes after a write operation

sf data query \
  --query "SELECT Name, Status__c, Customer_Name__c FROM Booking__c ORDER BY Name" \
  --target-org DinerAgent \
  --json

Option B — Agentforce Builder UI

Open Setup → Agentforce Agents in the sandbox org

Find Intelligent Diner Support Agent and click Open in Builder

Click Preview (top right). Enable Use Live Actions toggle to hit real Apex.

Type a message in the chat panel and observe the agent response and the Session Trace on the right — it shows which subagent was selected, which actions were invoked, and the LLM's reasoning.

Known: confirmation prompt appears twice on cancel. The subagent instructions and require_user_confirmation: True on the action both trigger a confirmation step. The first "yes" acknowledges the LLM's question; the second triggers the platform-native confirmation and actually calls the Apex action. This is a known redundancy — the action executes correctly on the second confirmation.

Deploy & Publish

The agent is currently deployed and working in preview mode. To publish and activate it for end users, run these commands in order from ~/Projects/CapstoneAgentforce.

Validate the agent script compiles without errors

sf agent validate authoring-bundle \
  --api-name Intelligent_Diner_Support_Agent \
  --json

Deploy the bundle and Apex classes (deploy all-source fails due to PermissionSet XML bug — deploy components individually)

sf project deploy start \
  --metadata "AiAuthoringBundle:Intelligent_Diner_Support_Agent" \
  --metadata "ApexClass:GetRestaurantDetails" \
  --metadata "ApexClass:CheckBookingAvailability" \
  --metadata "ApexClass:ModifyOrCancelBooking" \
  --metadata "ApexClass:CreateBooking" \
  --target-org DinerAgent \
  --json

Grant Apex class access to the Permission Set (must be done via Apex — XML deploy fails for this sandbox type)

// Run via: sf apex run --target-org DinerAgent
PermissionSet ps = [SELECT Id FROM PermissionSet WHERE Name = 'DinerAgent_Access' LIMIT 1];
for (String cls : new List<String>{'GetRestaurantDetails','CheckBookingAvailability','ModifyOrCancelBooking','CreateBooking'}) {
    ApexClass ac = [SELECT Id FROM ApexClass WHERE Name = :cls LIMIT 1];
    insert new SetupEntityAccess(ParentId = ps.Id, SetupEntityId = ac.Id);
}

Publish the authoring bundle — creates a versioned snapshot of the agent

sf agent publish authoring-bundle \
  --api-name Intelligent_Diner_Support_Agent \
  --json

Activate the agent — makes it available to end users

sf agent activate \
  --api-name Intelligent_Diner_Support_Agent \
  --json

Verify post-activation with the published agent (not authoring bundle)

sf agent preview start \
  --api-name Intelligent_Diner_Support_Agent \
  --json
Important: Every publish creates a permanent, immutable version number in the org. Only publish when you're ready — do not publish as part of the dev/test loop.
